# .ai-wiki/

LLM이 `outbox/`와 `.inbox/`의 본문을 직접 읽기 좋게 만드는 **인덱스 자리**. 점(`.`) prefix는 *사람이 직접 안 만지고 커맨드가 채우는 자동화 영역*이라는 뜻.

## 왜 outbox·inbox와 따로 있나

`outbox/`는 **사람이 다시 읽으려고** 쓴 서사. 본인 손맛이 살아 있어야 한다.
`.inbox/`는 **사람이 raw 자료를 던져둔** 자리. 사람 시점에서 다시 찾는 원본.
`.ai-wiki/`는 **LLM이 어느 페이지를 펼칠지 먼저 정하는** 카탈로그 자리.

같은 글을 두 번 적지 않는다. **본문은 한 자리(`outbox/` 또는 `.inbox/`)에만 있고**, `.ai-wiki/index.md`는 그 본문을 가리키는 메타·인덱스만 둔다.

> 안드레 카파시의 표현을 빌리면: 가장 강력한 노트는 future me · current me · LLM 셋 다를 동시에 만족시키는 노트다. `outbox/`·`.inbox/`는 사람을 위해, `.ai-wiki/index.md`는 LLM을 위해 같은 자료에 두 번째 시각(인덱스)을 박는다.

## v1 결정 — 본문 미러 없음

이전 버전은 `.ai-wiki/` 안에 본문을 복제했다. v1에서는 **인덱스만 두고 본문은 단일 진실원**으로 유지한다. 이유:

- 본문 한 자리 → drift 0
- `.ai-wiki/` 폴더가 가벼움 (index.md 중심)
- 카파시 원문의 wiki 정신("LLM이 만드는 자리")과 의미적 정합 — `.ai-wiki/`는 LLM 인덱스·메타·연결만, 본문은 사람 자리
- v1 단순성 정신 — 두 자리에 본문 두는 복잡성은 v2 검색 인프라 이후 고려

LLM은 `.ai-wiki/index.md` 한 장을 먼저 읽고, 필요한 페이지의 본문을 `Read outbox/{파일명}.md` 또는 `Read .inbox/{파일명}.md`로 펼친다.

## 두 type — raw / reflected

`.ai-wiki/index.md`의 페이지 한 줄에는 `type` 필드가 박힌다.

| type | 무엇을 가리키나 | 누가 채우나 | 인덱스에 박히는 정보 |
|---|---|---|---|
| `raw` | `.inbox/`의 raw 자료 (URL fetch · 텍스트 · 노션 export) | `/수집` | 한 줄 요약 (해석 X) + tags |
| `reflected` | `outbox/`의 사람-소화 글 | `/쓰기`가 placeholder, `/일일회고`가 본문 스캔으로 갱신 | claim 한 줄 + tags + 메타인지 + 열린 질문 |

**왜 type을 박는가** — raw 자료와 사람-소화 자료가 LLM 컨텍스트에 같이 들어오면, LLM이 두 종류를 동등하게 취급해서 raw를 사실로, 소화 글을 의견으로 오인할 수 있다. `type` 필드 하나로 "이건 내가 검토 안 한 자료" vs "이건 내가 결론까지 가본 자료"를 LLM도 본인도 명확히 구분한다.

## 어떻게 채워지나

- 사람이 직접 만지지 **않는다.**
- `/쓰기`가 outbox 새 글을 만들면 `index.md`에 reflected placeholder 한 줄 박는다 (claim은 사용자 의도 그대로, 메타·열린 질문은 비워둠).
- `/수집`이 raw를 던지면 `index.md`에 raw 한 줄 박는다.
- `/일일회고`가 매일 아침 outbox 전체를 스캔해서 reflected 항목의 claim·tags·메타인지·열린 질문을 최신화한다.
- 세 커맨드 모두 호출 끝에 `index.md`의 Log 섹션에 한 줄을 append한다.

## `index.md` — LLM이 가장 먼저 읽는 카탈로그

본문을 펼치기 전에 *어느 페이지를 펼칠지* 정하는 진입점. 카파시 LLM wiki 패턴의 핵심.

- **무엇이 들어가나** — 페이지 한 줄(`title · claim · tags · type` + reflected는 메타인지·열린 질문 두 줄) + 시간순 Log
- **누가 갱신하나** — `/쓰기`·`/수집`·`/일일회고` 자동
- **왜 이 한 장이 RAG를 대체하나** — 카탈로그가 LLM 컨텍스트 윈도우에 *통째로* 들어가서, 임베딩 유사도 검색 없이 *LLM이 직접 페이지를 고름*. 소스 100개·페이지 수백 개 규모까지 충분
- **사람이 만지지 않는다** — 다른 `.ai-wiki/` 파일과 동일

## 손대지 않는 이유

`.ai-wiki/`를 손으로 만지면 `outbox/`·`.inbox/`와 동기화가 깨진다. 본인이 만지고 싶은 글이 생기면 **`outbox/`로 가서 새 글로** 쓴다. `.ai-wiki/`는 LLM이 읽는 카탈로그지 작성판이 아니다.

## 청소 규칙

`index.md`가 너무 길어지면 그게 신호 — raw가 쌓이는데 사람이 소화하지 못한 것. `index.md` 자체를 청소하지 말고, `outbox/`에 한 편 써내거나 `/일일회고`를 호출해 패턴 한 줄로 압축한다.
