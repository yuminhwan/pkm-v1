# .ai-wiki Index

> 이 파일은 `/쓰기`·`/수집`·`/일일회고`가 자동으로 갱신한다. 사람이 직접 만지지 않는다.
> LLM이 Query 시 가장 먼저 읽는 카탈로그. 본문은 `outbox/`·`.inbox/`에 그대로 있고, 이 한 장이 *어느 페이지를 펼칠지* 결정한다.

## 페이지 (tag 기준 클러스터링)

> 아직 비어 있음 — `/쓰기`로 첫 outbox 글을 만들거나 `/수집`으로 raw 자료를 던지면 여기 한 줄씩 박힌다. tag 섹션은 새 태그가 등장하면 알파벳 순으로 추가된다.

## Log (시간 역순 append-only)

> 아직 비어 있음 — `/쓰기`·`/수집`·`/일일회고` 호출마다 한 줄씩 박힌다. 가장 최근 항목이 맨 위.

## 파일 한 줄 형식

```text
### {tag}
- [{title}]({link path}) — {claim 또는 한 줄 요약} #{tag1} #{tag2} [type: reflected|raw]
  - 메타인지: {반복된 패턴 한 줄}      ← reflected만, /일일회고가 채움
  - 열린 질문: {다음에 만날 질문 한 줄} ← reflected만, /일일회고가 채움
```

| 필드 | reflected (사람-소화 outbox 글) | raw (외부 자료·AI 답변) |
|---|---|---|
| `link path` | `../outbox/{파일명}` | `../.inbox/{파일명}` |
| `title` | 파일명에서 추출 (예: `지식관리의-함정`) | frontmatter `title` 또는 본문 첫 줄 |
| `claim` | outbox 첫 줄 주장 그대로 | 한 줄 요약 (본문 압축, 해석 X) |
| `tags` | 2〜4개 | 동일 |
| `type` | `reflected` | `raw` |
| 메타인지·열린 질문 | `/일일회고`가 매일 아침 본문 스캔해서 갱신 | 박지 않음 (raw에는 사람 소화 없음) |

## 갱신 정책

- `/쓰기`가 outbox 새 글을 만들면 페이지 섹션에 placeholder 한 줄 박는다 (claim은 사용자 의도 그대로, 메타·열린 질문은 비워둠).
- `/수집`이 raw를 던지면 페이지 섹션에 한 줄 박는다 (link path는 `../.inbox/...`).
- `/일일회고`가 매일 아침 outbox 전체를 스캔해서 reflected 항목의 claim·tags·메타인지·열린 질문을 최신화한다 (덮어쓰기).
- 같은 path가 두 번 등장하면 **덮어쓴다**. 인덱스는 *현재 상태*를 반영한다.
- Log는 절대 덮어쓰지 않는다 — 호출 이력의 단일 진실 공급원.

## 왜 본문 미러가 없는가 (v1 결정)

카파시 LLM wiki 패턴의 핵심은 **인덱스 한 장이 컨텍스트 윈도우에 들어가서, 임베딩 RAG 없이 LLM이 직접 페이지를 고른다**는 점이다. 본문 자체는 한 자리(`outbox/` 또는 `.inbox/`)에만 있어도 충분하다. v1에서는 본문 미러를 두지 않고 인덱스만 운영한다 — 단일 진실원 + 단순성 + 카파시 정신 정합.

LLM은 이 인덱스에서 페이지 한 줄을 보고 `Read outbox/{파일명}.md`로 본문을 펼친다. 소스 100개·페이지 수백 개 규모까지는 이 흐름이 충분.
