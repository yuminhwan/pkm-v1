# Kafka 파티션-브로커 동기화 및 토픽 관계

> 출처: Confluent Developer — Kafka for Beginners (https://developer.confluent.io/learn-kafka/architecture/)

---

## 1. Topics (토픽)

- 토픽은 **데이터베이스 테이블과 유사한 개념**으로, 동일 유형의 이벤트를 구성하는 단위
- **Append-only, immutable logs** — 추가된 이벤트는 변경되거나 삭제되지 않음
- 이벤트를 발행할 때 특정 토픽을 지정해야 하며, 소비 시에도 읽을 토픽 세트를 지정

---

## 2. Partitions (파티션)

- **데이터 분산의 단위**: 파티션은 "the main unit of storage for Kafka events"
- **병렬 처리의 단위**: 파티션은 "also the main unit of parallelism"
- 하나의 파티션은 기본적으로 "stored within a single broker in the Kafka cluster"
- Tiered Storage 지원 시 단일 브로커 이상의 용량으로 확장 가능

### 파티션의 이점
- 프로듀서가 여러 파티션에 동시에 기록하여 병렬 발행 가능
- 컨슈머가 서로 다른 파티션을 각각 읽음으로써 작업 부하를 분산
- 파티션이 하나뿐이면 컨슈머 인스턴스를 하나만 효과적으로 활용할 수 있음

---

## 3. Offset (오프셋)

- 파티션 내 각 이벤트에 부여되는 고유 식별자
- "monotonically increasing number and once it's given out, it's never reused"
- 이벤트는 오프셋 순서대로 저장되고, 컨슈머에게도 해당 순서로 전달됨
- 컨슈머가 처리 위치를 추적하는 데 활용됨

---

## 4. Brokers (브로커)

- Kafka 클러스터는 여러 브로커로 구성
- 파티션 데이터가 "typically stored within a single broker in the Kafka cluster"
- 클러스터 내 여러 노드에 파티션이 분산되어 상주
- Tiered Storage를 통해 단일 브로커의 용량 제한을 초과하는 데이터 저장 지원

---

## 5. Replication Factor & ISR (In-Sync Replicas)

### Replication Factor
- 각 파티션의 복제본 수
- 예: RF=3이면 3개 브로커에 동일 파티션이 복제됨
- 내결성과 고가용성을 보장

### ISR (In-Sync Replicas)
- 리더와 동기 상태를 유지하고 있는 복제본 목록
- ISR에서 벗어나면 해당 복제본은 장애로 간주되고 제외됨
- `replica.lag.time.max.ms`를 통해 동기화 임계값 관리

---

## 6. Consumer Groups (컨슈머 그룹)

- 같은 그룹에 속한 컨슈머들은 파티션을 나눠서 읽음
- **하나의 파티션은 한 그룹 내에서 단 하나의 컨슈머만 소비**
- 서로 다른 컨슈머 그룹은 동일 토픽을 독립적으로 소비 가능
- 그룹 내 컨슈머 수가 파티션 수를 초과하면 초과 컨슈머는 유휴 상태

---

## 7. Partition Assignment Strategy

| 전략 | 설명 |
|------|------|
| **RangeAssignor** | 파티션 번호를 컨슈머 수로 나누어 범위별 배정 |
| **RoundRobinAssignor** | 모든 파티션을 라운드 로빈 방식으로 순환 배정 |
| **StickyAssignor** | 기존 배정을 최대한 유지하면서 균형 있게 재배정 |
| **CooperativeSticky** | 점진적 리밸런싱을 지원하는 스티키 전략 |

## 8. 아키텍처 전체 구조

```
┌─────────────────────────────────────────────┐
│            Compute Layer (처리 계층)          │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ Producer  │ │ Consumer │ │ Kafka Streams│ │
│  │   API     │ │   API    │ │  / ksqlDB    │ │
│  └──────────┘ └──────────┘ └──────────────┘ │
│           ┌──────────────┐                   │
│           │ Kafka Connect│                   │
│           └──────────────┘                   │
├─────────────────────────────────────────────┤
│            Storage Layer (저장 계층)          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │Partition0│  │Partition1│  │Partition2│    │
│  │ Broker A │  │ Broker B │  │ Broker C │    │
│  └─────────┘  └─────────┘  └─────────┘     │
└─────────────────────────────────────────────┘
```

핵심 설계 철학: "the processing layer separated from the storage layer" — 저장 용량과 처리 용량을 독립적으로 확장 가능

## 데이터 흐름 요약

```
Producer → Topic → Partitions (Broker에 저장, Replication으로 복제) → Consumer Group (Partition Assignment를 통해 분배)
```
