# Kafka Under Replicated Partitions (URP) — 운영 중 발생 시 대처 가이드

## 1. 개념

Under Replicated Partitions는 **ISR(In-Sync Replicas) 목록에 있는 복제본 수가 replication factor보다 적은 파티션**을 의미합니다. 리더를 포함하여 동기화된 복제본이 충분하지 않은 상태로, 데이터 내구성과 가용성에 직접 영향을 줍니다.

- replication factor=3인데 ISR에 2개만 남으면 해당 파티션이 under-replicated 상태
- `kafka-topics.sh --describe` 출력에서 `Isr` 값이 `Replicas` 값보다 작으면 URP

### ISR (In-Sync Replicas)
- 파티션의 현재 동기 상태에 있는 복제본 집합
- 리더(Leader)와 일정 수준(**replica.lag.time.max.ms**) 내로 데이터를 따라잡은 팔로워만 ISR에 포함
- `min.insync.replicas` 이상이 ISR에 있어야 `acks=all` producers가 정상 동작

### Replication Lag
- 리더와 팔로워 간의 오프셋 차이 또는 시간 기반 지연
- `replica.lag.time.max.ms`가 핵심 튜닝 파라미터 (기본값: 30초)
- lag가 이 임계값을 초과하면 팔로워는 ISR에서 자동 제거

## 2. 발생 원인

| 원인 | 설명 | 신뢰 출처 |
|------|------|-----------|
| **브로커 다운/비정상** | 브로커 장애 시 해당 브로커의 복제본이 ISR에서 제외 | Apache Kafka 공식 문서 |
| **네트워크 지연/불안정** | 리더와 팔로커 간 복제 전송 지연으로 타임아웃 발생 | Confluent 문서 |
| **디스크 I/O 병목** | 팔로워 브로커의 디스크 쓰기 지연으로 복제 lag 발생 | Confluent 문서 |
| **CPU 과부하** | 팔로워 브로커 리소스 부족으로 복제 처리 밀림 | Apache Kafka 공식 문서 |
| **롤링 업데이트/재시작** | 컨트롤러의 파티션 재할당 중 일시적 URP | Apache Kafka 공식 문서 |
| **replica.lag.time.max.ms 초과** | 팔로워가 이 시간 내에 fetch하지 못하면 ISR에서 제거 | Apache Kafka 공식 문서 |
| **파티션 수 과도** | 브로커당 파티션이 과도하면 복제 지연 유발 | Confluent 권장사항 |
| **랙/AZ 미분산** | 동일 AZ에 리더+복제본 배치 시 동시 장애 위험 | Confluent 권장사항 |

## 3. 진단 방법

### JMX Metrics (핵심)
```
# URP 파티션 수 — 0보다 크면 즉시 주의
kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions

# ISR 삭제/추가 횟수
kafka.server:type=ReplicaManager,name=IsrShrinksPerSec
kafka.server:type=ReplicaManager,name=IsrExpandsPerSec

# 리더 선거 횟수
kafka.controller:type=KafkaController,name=LeaderElectionRateAndTimeMs

# 복제 레이턴시
kafka.server:type=ReplicaFetcherManager,name=MaxLag
kafka.server:type=ReplicaFetcherManager,name=ReplicationLag

# 오프라인 파티션 수
kafka.controller:type=KafkaController,name=OfflinePartitionsCount

# 컨트롤러 존재 여부
kafka.controller:type=KafkaController,name=ActiveControllerCount
```

### CLI 모니터링
```bash
# 전체 토픽 URP 확인
kafka-topics.sh --bootstrap-server localhost:9092 --describe --under-replicated-partitions

# 특정 브로커의 파티션 상태
kafka-leader-election.sh --bootstrap-server localhost:9092 --describe
```

### 모니터링 도구
- **Prometheus + Grafana**: `kafka-exporter` 또는 `Burrow` 활용
- **Confluent Control Center** / **Cruise Control**: 상업적/오픈소스 모니터링
- **Burrow**: 컨슈머 레이지 모니터링 특화
- **Datadog**: Kafka JMX 메트릭 수집 및 알림

## 4. 대처 방안

### 4.1 즉시 대응
1. **장애 브로커 확인** → 로그 및 시스템 상태 점검
2. `UnderReplicatedPartitions`가 0이 아니면 즉시 알림 발동
3. **컨트롤러 로그 확인**: 컨트롤러 브로커 로그에서 파티션 리더십 변경 이벤트 추적
4. **불필요한 리더 선거 억제**: `auto.leader.rebalance.enable=false` (긴급 시)

### 4.2 설정 튜닝
```properties
# 팔로워가 리더를 따라잡을 시간 (기본값 30초, 네트워크 환경에 맞게 증가)
replica.lag.time.max.ms=30000

# 최소 동기화 복제본 수 (acks=all 사용 시 최소 2 이상 권장)
min.insync.replicas=2

# 비동기화 replica가 리더 되지 않도록
unclean.leader.election.enable=false

# 복제 fetcher 스레드 수 증가
num.replica.fetchers=4

# 복제 fetch 최대 바이트 증가
replica.fetch.max.bytes=10485760
```

### 4.3 인프라 최적화
- **균등한 하드웨어**: 모든 브로커의 디스크 성능(SSD 권장), 네트워크 대역폭을 균일하게 유지
- **랙/Availability Zone 분산**: 리더와 복제본을 서로 다른 AZ에 배치하여 장애 도메인 분리
- **파티션 수 적절히 조절**: 브로커당 2,000~4,000개 이하 권장
- **OS 튜닝**: 파일 디스크립터 제한 증가, vm.dirty_ratio 튜닝, 스왑 비활성화

### 4.4 장기적 전략
- **Preferred Leader Election 자동화**: `auto.leader.rebalance.enable=true`로 리더 균형 유지
- **Cruise Control**로 자동화된 리밸런싱 구현
- **ISR shrink/expansion 추적**: ISR 크기 변화 추이를 모니터링하여 문제 예측
- **알림 임계값 예시**:
  - UnderReplicatedPartitions > 0 (1분 이상 지속) → Warning
  - UnderReplicatedPartitions > 0 (5분 이상 지속) → Critical
  - IsrShrinksPerSec > 0 → Warning

### 4.5 주의사항
- URP 상태에서는 `acks=all` Producer가 `NotEnoughReplicasException`을 받을 수 있으므로 애플리케이션 레이어에서 재시도 로직이 반드시 필요
- 롤링 업데이트 시 `controlled.shutdown.enable=true`로 정상 종료 보장
