# OpenSearch/Elasticsearch 핵심 개념

> 출처: Elasticsearch 공식 문서 (https://www.elastic.co/guide/en/elasticsearch/reference/current/indices.html)
> 참고: OpenSearch는 Elasticsearch의 fork로, 핵심 개념이 동일함

---

## 1. Cluster (클러스터)

- 클러스터는 여러 **노드(서버)**의 집합으로, 데이터를 함께 보유하고 인덱싱·검색 기능을 제공
- 클러스터 건강 상태는 `GET /_cluster/health`로 확인 가능
- 하나의 클러스터는 고유한 이름을 가지며, 기본값은 "elasticsearch"

---

## 2. Node (노드)

- 클러스터 내에서 단일로 실행 중인 OpenSearch/Elasticsearch 인스턴스
- 인덱싱, 검색, 데이터 저장 등의 작업을 처리
- `GET /_nodes`로 노드 정보, `GET /_nodes/stats`로 노드 통계 확인 가능

---

## 3. Index (인덱스)

- 유사한 특성을 가진 문서들의 **논리적 컨테이너**
- 관계형 데이터베이스의 "데이터베이스"와 개념적으로 유사
- 인덱스 API를 통해 개별 인덱스, 인덱스 설정, 별칭, 매핑, 인덱스 템플릿을 관리
- 인덱스 생성 시 **프라이머리 샤드 수**를 지정 (이후 변경 불가)

---

## 4. Shard (샤드)

- 인덱스의 **분할 단위** — 수평적 확장을 가능하게 하는 핵심 메커니즘
- 인덱스는 하나 이상의 **프라이머리 샤드**로 나뉘어짐
- 각 프라이머리 샤드는 클러스터의 서로 다른 노드에 분산됨

### 프라이머리 샤드 (Primary Shard)
- 각 문서는 정확히 하나의 프라이머리 샤드에 저장됨
- 문서 ID의 해시값을 라우팅 함수로 사용하여 특정 프라이머리 샤드에 할당
- 인덱스 생성 시 수 지정, **이후 변경 불가**

### 레플리카 샤드 (Replica Shard)
- 프라이머리 샤드의 **복제본**
- 장애 허용(High Availability) 제공 — 노드 장애 시 레플리카가 대체 가능
- 읽기 처리량 증가 — 검색 쿼리를 복제본에 분산하여 처리
- 수를 동적으로 변경 가능

---

## 5. Mapping (매핑)

- 문서와 필드가 어떻게 저장되고 인덱싱될지를 정의하는 **스키마**
- `GET /_mapping`으로 매핑 정의 확인, `PUT /_mapping`으로 필드 매핑 업데이트 가능

---

## 6. 데이터 흐름

```
1. 클라이언트가 문서를 노드에 전송
2. 코디네이팅 노드가 문서 ID 해시를 통해 적절한 프라이머리 샤드로 라우팅
3. 프라이머리 샤드에서 문서가 인덱싱됨
4. 레플리카 샤드로 복제
5. 모든 인-sync 복제본이 확인 응답을 보내면 성공 응답 반환
```

### 검색 흐름
- 검색 쿼리는 프라이머리·레플리카 샤드 모두로 **팬아웃(fan-out)**
- 코디네이팅 노드에서 결과를 집계하여 클라이언트에 반환
