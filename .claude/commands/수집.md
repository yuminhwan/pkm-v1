# /수집

raw 자료를 본인 저장소에 한 줄로 던지는 도구. 본문은 `.inbox/`에만 저장하고, `.ai-wiki/index.md`에는 한 줄 카탈로그만 박는다.

> **5원칙 1번(파일명 = H1 = 주장)의 예외 영역**: `/수집`이 담는 자료는 외부에서 가져온 raw 자료다 — 본인의 주장이 아니므로 키워드 슬러그(`{타임스탬프}-{핵심키워드}.md`)를 쓴다. 주장 슬러그 컨벤션은 `outbox/`(본인 손맛이 들어간 글)에만 적용된다. 이 구분이 무너지면 "주장이 있는 자리"와 "raw가 들어오는 자리"가 섞여 신뢰가 깨진다.

## 0단계 — 인자 파악

`/수집 <인자>` 전체를 본다. 인자에는 **본문 소스 + (선택) 추가 프롬프트**가 자유 텍스트로 섞여 들어올 수 있다.

**본문 소스 분리:**
- 인자 첫 토큰이 `http://`·`https://` → URL fetch 경로 (1단계)
- 그 외 → 텍스트 경로. 따옴표·줄바꿈으로 본문 소스와 추가 프롬프트 경계가 명확하면 분리, 모호하면 사용자에게 "무엇이 본문이고 무엇이 추가 요청인지" 한 번 짧게 확인
- 인자 없음 → 사용자에게 "무엇을 수집할까요? (URL 또는 텍스트 한 줄)" 묻고 답을 받는다

**추가 프롬프트 감지:**
- 본문 소스(URL·텍스트) 뒤에 자연어가 더 붙어 있으면 그게 **추가 프롬프트**
- 예시:
  - `/수집 https://example.com/article 어제 본 거랑 비슷한지 확인해줘` → URL + 추가 프롬프트 "어제 본 거랑 비슷한지 확인해줘"
  - `/수집 https://example.com/article` → URL만, 추가 프롬프트 없음
- 추가 프롬프트가 있으면 **1.5단계 활성화**

## 1단계 — 본문 확보

**URL 경로:** `WebFetch`로 URL을 가져온다. 본문을 markdown으로 변환하되, 광고·내비·푸터·소셜 버튼은 제거하고 헤드라인과 본문 단락만 남긴다. fetch 실패 시 사용자에게 알리고 텍스트 경로로 전환을 제안한다.

**텍스트 경로:** 입력 텍스트를 그대로 본문으로 쓴다. 어떤 가공도 하지 않는다.

## 1.5단계 — 관련 자료 찾기 (추가 프롬프트 있을 때만)

0단계에서 추가 프롬프트를 감지했으면 이 단계가 활성화된다. 추가 프롬프트가 없으면 **건너뛰고 2단계로**.

### 동작

1. `.ai-wiki/index.md` 전체를 읽는다.
2. 검색 키워드 잡기:
   - 1단계에서 확보한 본문의 핵심 키워드 2〜3개
   - 추가 프롬프트의 핵심 단어 1〜2개
3. 인덱스의 페이지 섹션에서 위 키워드와 매칭되는 항목을 찾는다. 매칭 우선순위:
   - 같은 tag 일치 (가장 강한 신호)
   - claim·요약·메타인지·열린 질문 본문에 키워드 등장
   - 같은 시기(최근 7일) 항목 가산점
4. **최대 3건**만 사용자에게 보여준다. 0건이면 "관련 자료를 찾지 못했습니다." 한 줄 보고하고 2단계로.

### 출력 형식

```
📚 관련된 자료를 .ai-wiki/index.md에서 찾았습니다:

1. [{slug}]({link path}) — {claim 또는 요약 한 줄} #{tag} [type: reflected|raw]
   왜 매칭: {tag 일치 / 키워드 X 등장 / 같은 시기}

2. [{slug}]({link path}) — {claim 또는 요약 한 줄} #{tag} [type: reflected|raw]
   왜 매칭: {짧은 근거}

3. [{slug}]({link path}) — {claim 또는 요약 한 줄} #{tag} [type: reflected|raw]
   왜 매칭: {짧은 근거}

추가 프롬프트에 대한 짧은 응답: {추가 프롬프트의 질문에 위 자료들을 근거로 한 줄 답변. 본문 펼치지 않고 인덱스 정보만으로}
```

### 제약

- **본문은 펼치지 않는다.** 인덱스 한 줄과 메타 두 줄만 보고 매칭. 본문이 필요하다는 판단이 들면 사용자에게 "본문을 펼쳐 확인할까요?" 한 번 묻고 분기.
- 매칭 후에도 **2단계 이후 흐름은 그대로 진행** — 새 raw 저장은 항상 수행된다. 1.5단계는 *함께 보여주기*일 뿐 저장 흐름을 가로채지 않는다.
- 매칭 항목을 새 raw의 `related:` 필드에 박는 자동화는 v1에서는 하지 않는다 (v1.5 작업). 사용자가 명시적으로 "관련으로 연결해줘"라고 하면 그때 4단계 frontmatter에 박는다.

## 2단계 — 파일명 정하기

타임스탬프는 `date +%Y-%m-%d-%H%M`. slug는 본문에서 추출:
- URL이면 페이지 title의 핵심 한 단어 (한글 가능)
- 텍스트면 첫 줄의 핵심 한 단어

최종 파일명: `{YYYY-MM-DD-HHMM}-{slug}.md`. 같은 분에 같은 slug가 있으면 slug 끝에 `-2`, `-3`을 붙인다.

## 3단계 — `.inbox/`에 원본 저장

`.inbox/{파일명}` 생성. frontmatter 최소:

```yaml
---
date: {YYYY-MM-DD-HHMM}
type: raw
source-url: "{원본 URL — 텍스트 입력이면 빈 문자열}"
title: "{문서 제목 또는 첫 줄}"
tags: [{본문에서 자연스럽게 뽑은 키워드 2-3개}]
---
```

그 아래 1단계에서 확보한 본문을 그대로 붙인다. 본문은 절대 요약·재작성하지 않는다.

## 4단계 — `.ai-wiki/index.md`에 한 줄 등록

LLM이 Query 시 가장 먼저 읽는 카탈로그에 이번 raw 자료 한 줄을 박는다. **본문 미러는 만들지 않는다** — v1에서는 본문 단일 진실원(`.inbox/`).

### 페이지 섹션 갱신

3단계 frontmatter의 `tags`에서 *주 tag*를 뽑아 `## 페이지` 아래 해당 섹션에 한 줄 추가. 같은 path가 이미 있으면 덮어쓴다 (보통 raw는 같은 path가 두 번 나오지 않지만 안전장치).

새 tag면 알파벳 순으로 새 섹션 만든다. 형식:

```text
### {tag}
- [{title}](../.inbox/{파일명}) — {본문 첫 단락에서 LLM이 뽑은 한 줄 요약} #{tag1} #{tag2} [type: raw]
```

raw의 한 줄 요약은 본문을 *압축*만 하고 *해석*하지 않는다. "이 글의 주장은 X다"가 아니라 "이 글은 X에 대해 다룬다" 같은 톤. **메타인지·열린 질문은 박지 않는다** — raw에는 사람 소화가 없으니까.

### Log 섹션 append

`## Log` 섹션 맨 위에 한 줄:

```text
- {YYYY-MM-DD HH:MM}: `/수집` ingest {파일명} ({원본 URL 또는 "텍스트 입력"})
```

### 인덱스 초기 골격 안내

`index.md`의 `> 아직 비어 있음 — ...` 안내 문구가 있으면 첫 항목 추가 시 같이 제거한다.

## 5단계 — 사용자에게 한 줄 보고

다음 형식으로 마무리한다. 1.5단계가 활성화됐다면 매칭 결과는 5단계 보고 *위에* 이미 출력돼 있으므로 여기서는 저장·갱신 보고만.

```
✓ .inbox/{파일명} 저장 ({원본 출처})
✓ .ai-wiki/index.md 갱신: +1 항목 · Log +1줄
```

1.5단계에서 매칭 자료가 나왔던 경우 한 줄 추가:

```
↳ 매칭된 관련 자료 {N}건은 위 1.5단계 출력 참고. 새 raw의 frontmatter에 연결로 박으려면 한 번 더 요청.
```

## 주의 사항

- `outbox/`는 절대 건드리지 않는다 — 사람용 영역. `/쓰기` · `/일일회고`만 outbox에 손댈 수 있다.
- `.ai-wiki/`에는 본문 파일을 만들지 않는다 — index.md 한 줄만 박는다 (v1 결정).
- 같은 URL을 두 번 `/수집`하면 두 번째는 타임스탬프가 달라서 별도 `.inbox/` 파일로 저장된다. 의도된 동작 — raw는 시간 흐름이 데이터다.
- index.md의 link path는 항상 `../.inbox/{파일명}` 형태 (`.ai-wiki/`에서 `.inbox/`로 가는 상대 경로).
- **1.5단계는 추가 프롬프트가 있을 때만 작동**한다. 단순 `/수집 <URL>`은 이전과 동일하게 빠르게 저장만 한다.
